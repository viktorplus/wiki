# Форматирование и стиль шпаргалок

Коротко: используйте последовательную структуру Markdown, короткие описания и стандартные теги, чтобы новые материалы легко читались и масштабировались.
#formatting, #linux, #guide
---

## Структура одной шпаргалки

1. **Заголовок:** `# команда — шпаргалка`.
2. **Введение:** 1–2 предложения о назначении команды.
3. **Теги:** сразу на следующей строке, например `#grep, #linux, #search`.
4. **Разделы:** `## Синтаксис`, `## Часто используемые опции`, `## Примеры`, `## Советы`, `## Ссылки`.
5. **Конец:** черта `---` и, при необходимости, список «могу добавить».

## Общие правила Markdown

- Старайтесь держать строки ? 80 символов, особенно в списках и описаниях.
- Используйте `` ` `` для inline-кода и блоков команды.
- Отделяйте блоки пустой строкой, чтобы избежать слипания списков.
- Вложенные списки делайте через два пробела и `-` (не номер).
- В кодовых блоках указывайте язык (обычно `bash`).

## Именование файлов/тегов

- Файл `команда.md` (например `ps.md`).
- Используйте англоязычные названия команд, без пробелов.
- Теги добавляйте в начале: `#command, #linux, #category`, насколько нужно.
- Внутри файлов можно ссылаться на другие шпаргалки: `см. 
`?

## Добавление примеров

- Пример должен быть самодостаточен: показывайте возможный ввод и ожидаемый выход.
- Для сочетаний с пайпами используйте `
`.
- Объясняйте, что делает каждой блок (например, `# найти логи за сутки`).

## Поддержание контента

- При появлении новой команды добавляйте в `README.md` и `formatting.md`.
- После изменения формата запустите `git diff` и проверьте, чтобы структура осталась читаемой.
- Используйте `todo` (Issue?)? yeah not necessary.

## Публикация и ссылки

- После создания привяжите файл к README или любому индексу.
- Можно вставлять ссылки на документацию (`man`, официальные сайты).
